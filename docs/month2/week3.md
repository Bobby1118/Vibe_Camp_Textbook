# Month 2: 유니티 엔진과 OOP의 결합

## Week 3: 📞 컴포넌트 통신망 (의존성과 핫라인)

지금까지 우리는 스크립트 한 덩어리 안에서 노는(계산하는) 법을 배웠습니다. 하지만 게임은 '플레이어 스크립트'가 '오디오 스크립트'에게, '몬스터 스크립트'가 'UI 스크립트'에게 끊임없이 **전화를 걸어 대화(통신)하는 과정**의 연속입니다. 
초보 코더나 AI는 이 전화를 무식하게 걸어대서 게임을 마비시킵니다. 이번 주차는 아티스트가 이 통신망 기지국을 완벽하게 다이렉트로 개통해 버리는 기술을 배웁니다.

### 📅 [Day 1] 컴포넌트 탐색의 공포 - 서울에서 김서방 찾기

#### 📖 1. 예습 (지옥의 행정 처리)
> **🏢 "주민등록번호 샅샅이 까보기" vs "다이렉트 핫라인"**
> 
> 총알 스크립트에서 피격 효과음 스크립트를 불러와 재생하고 싶습니다. 
> 
> *   **`GameObject.Find("SoundManager")`:** 서울 한복판에서 메가폰을 들고 "사운드 매니저님!!!" 하고 소리칩니다. 컴퓨터는 메모리에 적힌 10만 개의 오브젝트 이름을 위에서부터 전부 검색합니다. 프레임이 아작납니다.
> *   **`GetComponent<SoundManager>()`:** 같은 빈 게임 오브젝트(아파트)에 살고 있는 옆집 사람(컴포넌트)을 쾅쾅쾅 두드려 호출합니다. `Find`보단 백배 빠르지만, 총을 1초에 60번 쏠 때마다 `Update` 안에서 매번 이 문을 두드리면 여전히 컴퓨터가 느려집니다.

#### 🚨 2. 문제와 함정 (AI의 하드코딩)
```csharp
public class Bullet : MonoBehaviour 
{
    void Update() 
    {
        // 최악의 짓: 총알 날아가는 중(1초 60번)에 이름 검색!
        GameObject.Find("UI_Manager").GetComponent<UIManager>().ShowWarning();
    }
}
```

#### ✅ 3. 모범 답안 (캐싱 - 전화번호부 한 번만 등록하기)
```csharp
public class Bullet : MonoBehaviour 
{
    private UIManager uiManager; 

    void Start() 
    {
        // 씬 로딩 직후(Start) 시간이 남을 때 딱 1번만 검색해서 전화번호를 지갑에 저장! (캐싱)
        uiManager = GameObject.Find("UI_Manager").GetComponent<UIManager>();
    }

    void Update() 
    {
        // 이후 총알이 날아갈 때는 저장된 전화번호 버튼만 꾹! 렉 제로!
        uiManager.ShowWarning(); 
    }
}
```

#### 🗣️ 4. 지휘관의 프롬프팅
> **"야 AI! `Find`랑 `GetComponent` 같이 무겁게 탐색하는 쿼리(Query) 함수들 전부 다 `Update`나 액션 로직 안에서 뽑아 버려! 당장 `Start` (또는 `Awake`)로 끌어올려서 한번 룩업(Lookup)한 다음에 멤버 변수에 캐싱(Caching)해 둬!"**

### 📅 [Day 2] [SerializeField] 다이렉트 핫라인의 기적

#### 📖 1. 예습 (수동 배선 작업)
> **☎️ 검색조차 거부한다! 선을 직접 꽂기!**
> 
> `Start` 에서 검색 후 캐싱하는 것도 좋지만, 만약 디자인 팀에서 심심해서 오브젝트 이름을 `"UI_Manager"`에서 `"UI_System"`으로 바꿔버리면 어떻게 될까요?
> `GameObject.Find("UI_Manager")` 코드는 영원히 미아가 되어 끔찍한 에러를 뿜으며 터집니다. (이름 의존성의 재앙)
> 
> 아키텍터는 아예 코드에서의 탐색(`Find`)을 싹 다 지워버리고, **`[SerializeField]`로 빈 슬롯 구멍**만 뚫어둡니다. 그리고 유니티 에디터 화면으로 돌아와서 마우스 드래그 앤 드롭으로 A와 B를 **물리적으로 다이렉트 케이블(참조)**처럼 꽂아 버립니다!

#### ✅ 2. 핫라인 개통
```csharp
public class Bullet : MonoBehaviour 
{
    // [독해] "코드에서 이름 검색 안 함! 감독님이 알아서 내 인스펙터 구멍에 넣어줄 거임!"
    [SerializeField] private UIManager uiManager;

    void Update() 
    {
        uiManager.ShowWarning(); // 연결된 라인으로 0.0001초만에 즉시 통신
    }
}
```
**💡 디렉터의 쾌감**
이제 기획자가 UI 오브젝트 이름을 '시스템'이든 '매니저짱'이든 백 번을 뜯어고쳐도 에러가 나지 않습니다. 케이블이 직접 묶여 있기 때문입니다! 이것이 진정한 **컴포넌트 의존성 분리(Decoupling)**의 시작입니다.

### 📅 [Day 3] [RequireComponent] 방탄유리 설계

#### 📖 1. 예습 (깜빡 잊기와 강제성)
> **🛡️ "아 왜 스크립트 달았는데 터져요!"**
> 
> 내가 멋지게 만든 `CharacterMover` 스크립트. 내부에서는 `Rigidbody` (물리 중력) 컴포넌트를 사용(`GetComponent`)하도록 짰습니다.
> 그런데 초보 아티스트가 빈 큐브에 달랑 이 스크립트 하나만 드래그해서 넣고(중력 컴포넌트를 안 넣고) 플레이를 누릅니다. -> **오류 폭발! 게임 터짐!**
> 
> 이런 귀찮은 인적 오류(Human Error)를 원천 봉쇄하는 마법의 속성이 바로 `RequireComponent` 입니다.

#### ✅ 2. 방탄 설계 
이 스크립트를 달면, 유니티가 알아서 "어! 이 녀석은 `Rigidbody`가 필수네!" 하고 없으면 지가 알아서 같이 달아줍니다. 나중에 모르고 지우려 해도 "못 지움!" 이라며 알림창을 뿜으며 방어합니다.

```csharp
// [독해] "이 스크립트를 내 몸에 박고 싶으면 일단 무조건 Rigidbody부터 같이 달아라! 강제!"
[RequireComponent(typeof(Rigidbody))]
public class CharacterMover : MonoBehaviour 
{
    private Rigidbody rb;
    
    void Start() {
        rb = GetComponent<Rigidbody>(); // 100% 무조건 찾을 수 있음(안전보장)
    }
}
```

#### 🗣️ 3. 지휘관의 프롬프팅
> **"야 AI, 너 내 스크립트에서 `AudioSource` 가져다 컴포넌트로 쓰는데 왜 `[RequireComponent]` 안 달아놨냐? 협업자가 팀 프로젝트에서 오디오 소스 빼먹고 스크립트 달면 크래시 날 거 대비 안 하냐? 당장 필수 컴포넌트들 Require 달아서 에디터 안전장치 락 걸어!"**

### 📅 [Day 4] 싱글톤(Singleton)의 그림자 (맛보기)

#### 📖 1. 예습 (절대 권력의 시스템)
이 파트는 나만의 '통신 편의성'을 극대화시켜 주는 마법이지만 (결국 독이 되는) **싱글톤 패턴**을 아주 얕게 배웁니다.
`UIManager` 등 게임 전체를 주관하는 매니저들은 `[SerializeField]`로 몹 1만 마리에 일일이 선을 연결해 주기가 귀찮습니다.
그래서 매니저의 머리 위 하늘에 전 세계 어디서든 `UIManager.Instance` 라고만 호출하면 무조건 다이렉트 통신이 꽂히는 위성 안테나를 설치합니다. (상세 내역과 부작용 억제는 Month 4에서 다룹니다)

### 📅 [Day 5] 🕹️ 미니 프로젝트 & 핫라인 방탈출

#### 🎮 [미니 프로젝트] 에러 대환장 파티 고치기 
1. 유니티 씬에 오브젝트 2개를 만듭니다. 하나는 `Player`, 하나는 이름 오타가 난 `sound_manager(오타)` 라 짓습니다.
2. AI가 짜준 쓰레기 코드를 `Player`에 붙입니다.
```csharp
using UnityEngine;

public class PlayerFixing : MonoBehaviour 
{
    void Update() {
        // [버그유발] 매 프레임 이름으로 검색, 심지어 이름도 오타 남!
        GameObject.Find("SoundManager").GetComponent<AudioSource>().Play();
    }
}
```
3. **[지휘관의 미션]** 이 코드를 지우고, 대신 `[SerializeField]`를 사용해 오디오 소스 핫라인 구멍을 직접 뚫어 코드 수선을 진행하십시오!
4. 유니티로 돌아와 우측 인스펙터 슬롯에 `sound_manager`를 드래그해서 꽂고 렉 없이 무탈하게 구동되는 쾌감을 맛보십시오! (이름 오타가 났어도 케이블이 연결되어 완벽 작동합니다!)
