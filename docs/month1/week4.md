드디어 Month 1의 마지막, 그 악명 높은 **'객체 지향 프로그래밍(OOP)'**에 도착하셨습니다. 책에서는 이걸 철학적으로 너무 어렵게 풀어서 포기하게 만듭니다. 하지만 우리 아티스트들에겐 그저 **'붕어빵 공장 인테리어 디자인'**일 뿐입니다!

### 📅 [Day 1] Class의 본질 - 붕어빵 틀

#### 📖 1. 예습 (허공에 재료 뿌리기 vs 무쇠 틀 짜기)
> **🐟 허공의 밀가루(`Variables`) vs 무쇠 붕어빵 틀(`Class`)**
> 
> 고블린 100마리를 만들 때마다 프로그래머가 엑셀의 빈칸을 채우듯 `goblin1_hp`, `goblin2_hp` 이름을 100개 만들며 흩뿌린다? 이건 밀가루 반죽을 허공에 주먹으로 던지는 꼴입니다.
> 
> 위대한 우리 디렉터들은 허공에 데이터를 뿌리지 않습니다. 쇳물로 아주 단단한 **무쇠 붕어빵 틀(`Class`)** 하나를 1번만 선언해 둡니다. 그리고 그 빈 공간에 슈크림을 넣으면 슈크림 붕어빵(`Instance/객체`)이 뿅 튀어나오고, 팥을 넣으면 팥 붕어빵 1개가 메모리에 뿅 복사되어 튀어나옵니다. 1초에 수만 개를 찍어내는 공장의 시작입니다.

#### ✅ 2. 붕어빵 틀 독해
```csharp
// [문장 1 독해] 무쇠 붕어빵 틀 강제 선언! (메모리에는 아직 실체가 없음)
public class Monster 
{
    public float hp;
    public string type;
}
```

**🔍 코드 외계어 독해 가이드**
*   👉 **코드 공식:** `[권한] class [틀의이름] { 속재료들 }`
*   🗣️ **소리내어 읽기:** *"누구나 가져다 쓸 수 있게(`public`), 지금부터 `Monster`라는 이름의 새로운 붕어빵 주물 틀(`class`)의 설계도를 짤 거다! 이 틀로 구워낸 빵들은 무조건 체력(`hp`)과 속성(`type`)이라는 칸이 존재해야 해!"*
*   **❓ 설계도(`class`)만 만들면 화면에 몬스터가 등장하나요?**
    *   아닙니다! 도면만 그렸다고 공터에 아파트가 생기진 않는 것처럼, 붕어빵 기계 도면만 만들었으므로 아직 세상엔 진짜 팥 붕어빵이 **"존재(Instance)하지 않습니다."** 우리는 `new`라는 명령어로 빵을 구워달라고(찍어달라고) 컴퓨터에게 명령을 내려야만 비로소 세상에 물건이 뚝딱 탄생합니다!

### 📅 [Day 2] 멤버(Member) 구성원 - 영양성분표와 뜨거운 김

#### 📖 1. 예습 (물건의 2가지 요소)
세상의 모든 붕어빵(클래스 틀로 구워진 물건들)은 딱 2가지를 몸 안에 지니고 태어납니다. 
1.  **영양성분표 (명사 = 변수):** 무게 150g, 당도 50, 재료 팥. 수치로 된 고정 데이터.
2.  **행동 규범 (동사 = 함수):** 뜨거운 김 내뿜기, 바삭바삭 소리 내기. 이 객체가 혼자서 할 수 있는 스스로의 액션.

```csharp
public class Monster 
{
    // 1. 명사 영역: 영양 성분표 (Member Variable)
    public float hp = 100f;
    
    // 2. 동사 영역: 이 녀석이 직접 스스로 할 수 있는 행동 (Member Method)
    public void Attacked() 
    {
        hp -= 10;
        Debug.Log("으악! 나 맞았음!");
    }
}
```
**💡 디렉터의 통찰**
이 클래스 틀로 `오크`와 `고블린`을 찍어내면, 체력(`hp`) 지갑은 오크 지갑과 고블린 지갑 따로따로 놀지만, **`Attacked()`라는 맞는 행동 자체는 동일한 룰북(틀)을 따르게 됩니다**. 이것이 유지보수가 소름 돋게 편한 이유입니다!

### 📅 [Day 3] 생성자(Constructor) - 출생신고서 쓰기

#### 📖 1. 예습 (고아 몬스터 버그 사건)
> **👶 출생 직후 강제 룰!**
> 
> 고블린을 생성 도면으로 찍어냈습니다. 근데 깜빡하고 "이 녀석은 체력이 100이야"라는 초기 데이터 세팅을 안 해줬습니다! 이 고블린은 체력이 뭔지도 모르는 **초기화 안 된 고아 상태**로 태어납니다. 한 대 때리면 유니티는 체력이라는 수치가 비어있기 때문에 사상 최악의 `NullReferenceException` 오류를 뿜고 게임을 통째로 강제 종료시킵니다.
> 
> 이런 끔찍한 사태를 막으려면 어떻게 해야 할까요? **생성자(Constructor)**라는 녀석을 달아주면 됩니다. 이것은 붕어빵이 뿅 튀어나오는 바로 그 첫 0.1초 순간에 **무조건!! 강제로 1번 무조건 실행되는 출생신고 세팅 로직**입니다.

#### ✅ 2. 강제 출생신고 독해
```csharp
public class Monster 
{
    public float hp;

    // [문장 1 독해] 클래스 이름과 똑같이 쓴 이 녀석이 바로 강제 출생신고서(생성자)
    public Monster(float initHp) 
    {
        hp = initHp; // 이 녀석이 태어나는 순간, 무조건 외부에서 받은 숫자로 내 피를 박아라!
    }
}

// 부르는 스크립트 쪽
// Monster 고아 = new Monster(); // <- 이제 괄호 안 비우고 그냥 태어나려 하면 에러 뿜음! (완벽한 방어)
Monster 오크 = new Monster(500f); // 500이라는 초기 수치를 꼭 들려보내며 태어나게 강제!
```

### 📅 [Day 4] Static (정적) - 마법의 공용 팁(Tip) 박스

#### 📖 1. 예습 (개인 지갑이냐? 광장 팁 박스냐?)
> **💰 모두가 함께 보는 신비한 광장 팁 박스**
> 
> 몬스터 1,000마리의 체력 지갑은 자기 바지 뒷주머니에 각각 단독으로 있어야 합니다. 고블린을 때렸다고 오크 피가 깎이면 안 되니까요. (1주 차의 종이접시 Value 개념이죠!)
> 
> 그런데, **"현재까지 죽은 전체 몬스터 숫자"**를 세야 한다면 어떨까요? 이 카운터를 1,000마리가 각자 지갑에 들고 다니다가 계산하려면 1,000개의 지갑을 다 열어서 칠판에 적어야 됩니다.
> 이럴 때, 붕어빵 마을 전체 광장 정중앙에 거대한 **공용 금고 상자(`static`)**를 하나 탁! 내려놓습니다. 이 변수 앞에는 `static`이라는 특별한 라벨이 붙습니다. 
> 
> 모든 몬스터들은 죽을 때마다 개인 지갑 안 보고 곧바로 광장을 바라보며 **광장 금고(`static`)에 동전을 하나씩 툭툭 넣습니다.** 1,000마리가 언제든 한눈에 쳐다보고 공유하는 단 하나의 절대 데이터! 이것이 `static`의 무시무시한 공유 마법이자 거대한 메모리 절약법입니다.

#### ✅ 2. 팁 박스(Static) 독해
```csharp
public class Monster 
{
    // 개인 호주머니 (고블린과 오크가 서로 값이 다름)
    public float myHp = 100f;

    // [문장 1 독해] 몬스터 마을 광장에 놓인 유일한 공용 팁 박스
    public static int TotalDeadCount = 0;

    public void Die() 
    {
        myHp = 0; // 내 주머니 피는 바닥나고
        Monster.TotalDeadCount += 1; // 몬스터가 공용 광장 박스의 숫자를 스스로 하나 올림!
    }
}
```

*   **❓ 그러면 전부 다 편하게 `static`으로 광장에 까버리면 안 되나요?**
    *   **"전역 변수의 함정(Week 2 Day 3)"**을 떠올려 보십시오. 편하다고 지갑이고 일기장이고 모든 데이터를 뻥 뚫린 광장(static)에 내다 버려두면? 지나가던 엉뚱한 스크립트가 발로 차서 값이 오염됐을 때 대체 누가 발로 찼는지 영원히 범인을 잡을 수 없게 됩니다. `static`은 정말 전 우주가 공유해야 하는 '규칙'이나 '매니저 수치'에만 극히 제한적으로 써야 하는 '절대 반지'와 같습니다.
