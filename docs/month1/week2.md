### 📅 [Day 1] 함수(Method)와 재사용성 - 복붙의 늪 탈출

#### 📖 1. 예습 (수작업 노가다 vs 마법의 도장)
> **🖨️ 똑같은 편지 100장 쓰기(복붙) vs 마법 도장 선언하기(`Method`)**
> 
> 고블린 100마리에게 '공격할 때 소리를 내고, 빨갛게 변해라' 라는 명령을 내립니다. AI를 감시하지 않으면 AI는 편지 100장을 일일이 손으로 써서 고블린 100마리 스크립트에 **그대로 복붙해 둡니다.** 
> 나중에 아트 디렉터가 "소리가 너무 큰데 볼륨 좀 줄여" 라고 지시하면 어떻게 될까요? 프로그래머는 100개의 스크립트를 하나하나 열어서 볼륨 수치를 고치는 생지옥을 경험합니다.
> 
> 반면, 아키텍터는 **"공격 행동"이라는 마법의 도장(함수)을 딱 하나 만듭니다.** 고블린 100마리에게는 그 도장을 꾹꾹 찍어주기만(호출) 합니다. 볼륨을 수정할 때는 100마리를 고칠 필요 없이, 원본 도장의 잉크만 한 번 수정하면 100마리가 동시에 변합니다. 이것이 함수의 **재사용성**입니다.

#### 🚨 2. 문제와 함정 (AI의 쓰레기 코드)
AI에게 고블린과 오크가 죽는 코드를 짜달라고 했더니, 끔찍하게 중복되는 코드를 복사해서 흩뿌렸습니다.

```csharp
public class Goblin : MonoBehaviour 
{
    void Update() {
        if (hp <= 0) {
            // 똑같은 로직이 복붙되어 있음!
            PlaySound("DieSound");
            CreateEffect("Blood");
            Destroy(gameObject);
        }
    }
}

public class Orc : MonoBehaviour 
{
    void Update() {
        if (hp <= 0) {
            // 여기도 토씨 하나 안 틀리고 또 복붙되어 있음!
            PlaySound("DieSound");
            CreateEffect("Blood");
            Destroy(gameObject);
        }
    }
}
```
**🤔 디렉터의 분노 지점**
만약 몬스터가 죽을 때 이펙트를 "바람 이펙트"로 바꾸고 싶다면? 고블린, 오크, 드래곤, 슬라임 스크립트를 수백 개 열어보며 각각 흩어져 있는 코드를 일일이 고쳐야 합니다. 단 하나라도 빼먹으면 오크 혼자서 피를 흘리는 버그가 터집니다.

#### ✅ 3. 모범 답안 (도장 파기)
공통된 행동은 무조건 하나의 덩어리로 묶어서 "원본 도장(함수)"으로 만들어야 합니다.

```csharp
public class MonsterBase : MonoBehaviour 
{
    // [문장 1 독해] 마스터 도장 만들기 (재사용 가능한 함수 블록 선언)
    public void HandleDeath() 
    {
        PlaySound("DieSound");
        CreateEffect("Wind"); // 여기서 한 번만 바꾸면 모든 몹이 바람을 내뿜음!
        Destroy(gameObject);
    }
}
```

#### 🗣️ 4. 지휘관의 프롬프팅
> **"AI야, 몬스터 죽는 코드를 왜 다 스크립트마다 하드코딩으로 복붙해 놨어? 나중에 유지 보수하려고 날밤 깔 셈이야? 당장 중복 로직 묶어서 `HandleDeath()` 라는 공통 함수(도장) 뽑아내!"**

### 📅 [Day 2] 파라미터(매개변수) - 동적 자재 투입구

#### 📖 1. 예습 (통합 자재 투입구 비유)
> **🏭 자판기 100대 놓기 vs 동전 구멍(파라미터) 뚫린 만능 자판기 1대**
> 
> "빨간 포션 마시기", "파란 포션 마시기" 기능이 필요합니다. 초보 코더는 빨간 포션용 함수 조이스틱 1개, 파란 포션용 함수 조이스틱 1개를 별도로 만듭니다. 아이템이 1,000개면 함수도 1,000개가 생겨서 스크립트가 찢어집니다.
> 
> 똑똑한 디렉터는 **'포션 마시기'**라는 기계 딱 1대만 만들고, 그 기계 위에 **동전 구멍(파라미터)**을 하나 뚫습니다. 그리고 구멍에 빨간색 포션 데이터, 파란색 포션 데이터만 휙휙 던져 넣으라고 지시하죠. 이 동전 구멍 역할을 하는 것이 바로 `( )` 괄호 안에 들어가는 매개변수(파라미터)입니다.

#### 🚨 2. 문제와 함정 (AI의 쓰레기 코드)
```csharp
public class Player : MonoBehaviour 
{
    // 끔찍한 하드코딩 함수 도배
    public void HealRedPotion() { hp += 50; }
    public void HealBluePotion() { mp += 20; }
    public void HealGreenPotion() { poison = 0; }
}
```

#### ✅ 3. 모범 답안과 외계어 독해
```csharp
public class Player : MonoBehaviour 
{
    // [문장 1 독해] 구멍 뚫린 만능 자판기 선언
    public void HealPotion(int amount, string type) 
    {
        if (type == "HP") hp += amount;
        else if (type == "MP") mp += amount;
    }
}
```

**🔍 잠깐! 코드 외계어(C# 문법) 독해 가이드**

**[문장 1]** `public void HealPotion(int amount, string type)`
*   👉 **코드 공식:** `[권한] [리턴(영수증)여부] [액션이름] (구멍1규격 구멍1이름, 구멍2규격 구멍2이름)`
*   🗣️ **소리내어 읽기:** *"누구나 버튼을 누를 수 있고(`public`), 영수증 뱉을 필요 없는(`void`) '물약 먹기(`HealPotion`)' 기계 작동! 단, 기계를 돌릴 때 구멍 2채널에 재료를 꼭 넣어줘야 돼. 첫 번째 구멍은 숫자형(`int`) 회복량(`amount`)을, 두 번째 구멍에는 문자열(`string`) 종류(`type`)를 던져 넣어라!"*

*   **❓ 그냥 함수 100개 따로 만들면 직관적이고 편하지 않나요? 괄호(구멍) 뚫기가 어색해요.**
    *   초반엔 편해 보여도 재앙의 시작입니다. 나중에 기획이 바뀌어 "물약을 마실 땐 무조건 빛나는 이펙트를 추가해!"라는 지시가 내려오면, 함수 100개를 다 열어서 이펙트 코드를 추가해야 합니다.
    *   파라미터 구멍 하나 뚫어둔 만능 기계를 쓰면, 기계 안쪽에 이펙트 코드 딱 한 줄만 추가하면 1,000가지 물약 전체에 이펙트가 한방에 적용됩니다. 

#### 🎯 3-1. 보너스 트랙: 그럼 기계를 어떻게 작동시키나요? (함수 호출)
만능 통합 기계(`HealPotion`)를 성공적으로 만들었으니, 이제 인벤토리 아이템 창에서 **실제로 기계의 구멍에 동전(파라미터 재료)을 집어넣어 작동시키는 모습**을 볼까요?

```csharp
public class Inventory : MonoBehaviour 
{
    // 우리가 통제할 플레이어 기계가 누구인지 타겟을 잡습니다.
    public Player myPlayer; 

    // 인벤토리 화면에서 [빨간 포션 UI 버튼]을 마우스로 클릭할 때!
    public void OnClickRedPotion() 
    {
        // 버튼에 숨겨져 있던 동전 2개(50과 "HP")를 기계 구멍에 휙 던져 넣습니다.
        myPlayer.HealPotion(50, "HP");
    }

    // 인벤토리 화면에서 [파란 포션 UI 버튼]을 마우스로 클릭할 때!
    public void OnClickBluePotion() 
    {
        // 따로 파란 전용 기계를 만들 필요 없이, 똑같은 기계에 동전 규격만 다르게 던집니다!
        myPlayer.HealPotion(20, "MP");
    }
}
```
**💡 아키텍터의 쾌감 포인트**
보이시나요? `HealRedPotion()`, `HealBluePotion()` 처럼 무식하게 긴 코드를 부를 필요가 영영 사라졌습니다. 그저 똑같은 이름의 `HealPotion()` 기계를 가져다 대고, 괄호 `( )` 사이에 던져줄 재료값 하나만 딱딱 바꿔치기하면 유니티가 알아서 내부 처리를 합니다. 코드가 1/100로 압축되는 마법입니다!

*   **❓ 잠깐! 파란 포션을 썼다는 걸 컴퓨터가 대체 어떻게 알죠? 코드에는 '파란 포션'이라는 단어가 아예 없는데요?**
    *   지휘관님, 논리의 끝을 파고드는 무서운 통찰입니다! 정답은 **"컴퓨터는 파란 포션이 뭔지 모른다"** 입니다.
    *   유니티 에디터 화면을 상상해 보세요. 파란 포션 그림(UI 버튼) 뒤에는 개발자가 몰래 **`(20, "MP")`** 라는 동전 세트를 쥐여주었습니다. 플레이어가 파란 포션 그림을 마우스로 '딸깍' 클릭하는 순간, 버튼이 자신이 쥐고 있던 동전 두 개를 `HealPotion` 기계에 맹목적으로 집어넣습니다.
    *   즉 기계는 "파란 포션이군!" 하고 알아채는 게 아니라, 기계 안에 "MP"라는 글자와 "20"이라는 숫자가 굴러들어오니까 그냥 조건판에 맞춰서 MP를 20 올려줄 뿐입니다. 이것이 바로 디자인(UI)과 로직(코드)이 파라미터를 통해 대화하는 원리입니다!

#### 🗣️ 4. 지휘관의 프롬프팅
> **"AI야, 무기마다 공격 함수를 다 분리해서 50개 써붙여 놨어? 쓸데없는 하드코딩 그만하고 당장 데미지 수치값을 파라미터 매개변수(Argument)로 넘겨받는 통합 함수 하나로 합쳐!"**

### 📅 [Day 3] 범위 스코프(Scope) - 공중 화장실의 저주

#### 📖 1. 예습 (공중 화장실 비유)
> **🚽 서울역 공중 화장실(전역 변수) vs 내 방 화장실(지역 변수)**
> 
> 디렉터님, 계산용 임시 데이터를 어디에 선언하시겠습니까? 스크립트 최상단(`class` 바로 밑)에 선언하는 것은 그 데이터를 **자물쇠 없는 서울역 화장실(전역 변수)**로 만드는 것과 같습니다. 사운드 시스템, UI 시스템, 몬스터 스크립트 등 누구나 똥칠을 하고 도망갈 수 있어서, 에러가 터졌을 때 용의자를 좁힐 수가 없습니다.
> 
> 반면, 계산용 변수를 함수 중괄호 `{ }` 내부에 선언하면 그건 **내 방 잠긴 화장실(지역 변수)**이 됩니다. 오직 그 행동(함수)이 켜졌을 때 잠깐 존재했다가, 행동이 끝나면 화장실 자체가 완전히 증발하므로 세상에서 가장 안전하고 위생적인 데이터 폭파 기술입니다.

#### 🚨 2. 문제와 함정 (AI의 쓰레기 코드)
```csharp
public class Calculator : MonoBehaviour 
{
    // 심각한 설계 결함: 이번 턴만 쓸 임시 계산 값을 화장실 한복판에 전역으로 판 놈!
    private float tempDamage; 

    public void AttackGoblin() {
        tempDamage = 100f * 1.5f; 
        hp -= tempDamage;
    }
}
```
**🤔 이게 왜 지옥을 부르나요?**
`tempDamage`라는 이름만 봐도 함수 안에서 1초 쓰고 버릴 데이터인데, 스크립트 꼭대기에 선언해서 영원히 메모리를 차지하게 만들었습니다. 만약 누군가 다른 함수에서 저 `tempDamage` 값을 재활용해서 읽게 된다면, 고블린을 쳤던 옛날 데미지 찌꺼기가 갑자기 드래곤한테 들어가 버리는 전대미문의 괴현상(오류)이 터집니다.

#### ✅ 3. 모범 답안 (중괄호 방음벽 치기)
1회성 데이터는 무조건 함수 내부에 선언해서 중괄호 `{ }`가 끝날 때 같이 소각되어야 합니다.

```csharp
public class Calculator : MonoBehaviour 
{
    public void AttackGoblin() {
        // [문장 1 독해] 이 중괄호 안에서만 잠깐 생겼다가 흔적도 없이 완전히 박살 날 지역 변수(Local Scope)
        float tempDamage = 100f * 1.5f; 
        hp -= tempDamage;
    } // <- 이 선을 넘는 순간 tempDamage 메모리는 소각로에서 완전히 불타 없어짐
}
```

#### 🗣️ 4. 지휘관의 프롬프팅
> **"AI야, 함수 내부에서 잠깐 계산하고 버릴 쓰레기 임시 변수를 왜 클래스 전역 스코프(Global Scope)에 띄워놨어? 나중에 변수값 유지 버그 터트리려고 환장했나, 당장 싹 다 함수 내부 지역 변수(Local Scope)로 잡아넣어!"**

### 📅 [Day 4] 리턴(Return) - 물리적 배달 사고 탈출

#### 📖 1. 예습 (배달 심부름 비유)
> **🏃‍♂️ 입으로만 대답하는 알바생(`void`) vs 거스름돈을 손에 쥐여주는 알바생(`return`)**
> 
> 1주 차에 우리는 결과를 안 돌려줘도 되는 `void`(행동만 하고 영수증 버리기)를 배웠습니다. (스피커 소리 틀기 등).
> 하지만 "현재 몬스터의 전체 방어력을 계산해 줘" 같은 명령을 `void`로 시키면 대형사고가 납니다. 컴퓨터는 백과사전을 뒤져서 방어력 1,500을 계산하지만, 영수증(데이터)을 뽑아주지 않기 때문에 정작 체력바 UI는 방어력이 얼만지 몰라 화면에 띄울 수가 없습니다.
> 
> 그래서 계산 데이터가 필요할 때는 행동 명령을 내릴 때 **목표하는 결과물 규격(예: `int`)**을 지정하고, 함수 안에서 `return`을 사용해 결과물을 디렉터의 양손에 철컥 쥐여주어야 합니다.  

#### 🚨 2. 문제와 함정 (AI의 쓰레기 코드)
```csharp
public class Setup : MonoBehaviour
{
    public int finalDamage; // 전역 변수 배설... 최악의 선택

    // 계산만 허공에 하고 외부망에 똥칠하는 void
    public void CalculateDamage(int atk, int def) 
    {
        finalDamage = (atk * 2) - def;
    }
}
```

#### ✅ 3. 모범 답안과 외계어 독해
계산을 했으면 깔끔하게 결괏값 포장 박스로 뱉어내게 강제해야 합니다.

```csharp
public class Setup : MonoBehaviour
{
    // [문장 1 독해] 행동을 끝내고 나에게 정수(int) 영수증을 건네주는 기계
    public int CalculateDamage(int atk, int def) 
    {
        int tempCal = (atk * 2) - def;
        return tempCal; // [문장 2 독해] 최종 생산된 캔을 기계 밖으로 뱉기!
    }
}
```

**🔍 잠깐! 코드 외계어(C# 문법) 독해 가이드**

**[문장 1]** `public int CalculateDamage(...)`
*   👉 **코드 작성 공식:** `[권한] [돌려받을물건규격] [액션이름] (구멍들)`
*   🗣️ **소리내어 읽기:** *"누구나 누를 수 있고(`public`), 다 끝나면 나한테 숫자형 데이터(`int`)를 영수증으로 돌려줘야 되는 기계 선언! 이름은 데미지 계산기!"*

**[문장 2]** `return tempCal;`
*   👉 **코드 작성 공식:** `return [선언했던규격의데이터];`
*   🗣️ **소리내어 읽기:** *"기계 내부에서 계산 싹 다 끝났지? 그럼 마지막에 계산된 그 결과물을 기계 밖으로 뱉어내고(`return`) 셔터 문 닫아라!"*

#### 🗣️ 4. 지휘관의 프롬프팅
> **"AI야, 수학 계산하는 단순 로직 짜달라니까 왜 `void`로 깎고 전역 변수에 결과값을 쑤셔놓고 있어? 당장 계산식 책임 분리해서 데이터 타입 `return` 하는 순수 함수(Pure Function)로 분리해!"**

#### 🎮 [미니 프로젝트] 동전 먹는 만능 자판기 조립
1.  **준비:** 유니티 화면에 `UI -> Button` 2개를 만들고, 각각 '빨간물약', '파란물약'이라고 이름을 짓습니다.
2.  **생성:** `VendingMachine` 스크립트를 만들고 빈 게임오브젝트에 붙입니다.
3.  **코드 작성:** 딱 하나의 통합 함수만 만듭니다.
```csharp
using UnityEngine;

public class VendingMachine : MonoBehaviour 
{
    // 기계의 유일한 작동 구멍(파라미터)
    public void DrinkPotion(string potionType) 
    {
        Debug.Log(potionType + " 포션을 마셨습니다! 캬~");
    }
}
```
4.  **쾌감의 순간:** 유니티 버튼의 `OnClick()` 이벤트에 방금 만든 요술 기계를 끌어다 넣습니다. 
    *   빨간물약 버튼에는 **"HP"**라는 글자를 타건해 넣고,
    *   파란물약 버튼에는 **"MP"**라는 글자를 타건해 넣습니다.
    *   **플레이(▶) 버튼을 누르고:** 게임 창에서 각 버튼을 눌러보십시오. 함수는 딱 1개뿐인데, 누르는 버튼(UI)에 따라 콘솔 창에 각각 다른 글씨가 출력됩니다! 이것이 디자인과 로직이 대화하는 아키텍처의 맛입니다.

public class ItemSystem : MonoBehaviour
