### 📅 [Day 1] 값(Struct)과 참조(Class) - 프레임 드랍과 쌍둥이 저주

![무거운 유리접시(Class)가 쌓이는 혼란과 종이 포장지(Struct)가 빛으로 증발하는 메모리 효율 비교 이미지](/assets/images/class_vs_struct.png)

#### 📖 1. 예습 (초등학생도 이해하는 '식당과 쪽지' 비유)
왜 수많은 코딩 책들은 이 쉬운 걸 외국어처럼 꼬아서 설명할까요? 디자이너의 언어로 번역하면 이렇게나 직관적인데 말이죠!

> **🍔 첫 번째 차이: 유리접시(청소기 렉) vs 종이접시(증발)**
> 상상해 보세요. 맥도날드에 손님이 10,000명 몰려옵니다. 
> 햄버거를 **무거운 유리접시(`class`)**에 담아 주면, 다 먹은 접시가 테이블에 끝도 없이 쌓입니다. 결국 햄버거를 더 팔려면 **영업을 물리적으로 멈추고 빗자루 아저씨(Garbage Collector)가 와서 우당탕 치워야 합니다.** 이때 게임 화면이 멈추는 '렉(프레임 드랍)'이 발생합니다.
> 반면, 햄버거를 **가벼운 종이 포장지(`struct`)**에 싸서 주면, 다 먹자마자 주머니에 넣었다가 스스로 쓰레기통에 쏙 버립니다. 빗자루 아저씨가 올 필요도 없고, 영업(게임)이 멈추지도 않습니다! 

> **👯‍♂️ 두 번째 차이: 쪽지의 저주 (참조형 얕은 복사)**
> 더 끔찍한 차이가 있습니다. `class` 데이터를 다른 녀석에게 복사해 달라고하면, 햄버거를 새로 만들어주지 않고 **'그 햄버거가 VIP칸 몇 번 테이블에 있는지 적힌 쪽지(주소)'**만 복사해 줍니다! (참조/Reference)
> 만약 고블린 A의 햄버거(체력) 쪽지를 B에게 줬다고 칩시다. 플레이어가 고블린 A를 한 대 때려서 햄버거를 까먹으면, 같이 공유하고 있던 쪽지를 들고 테이블을 찾아간 B의 햄버거도 동시에 사라져버립니다! (AI가 가장 많이 터트리는 공포의 '변수 공유 버그')
> 하지만 `struct`(종이 포장지)는 복사해달라고 하는 순간 그 자리에서 **햄버거 세트 자체를 통째로 하나 더 만들어서(완벽한 복제본/Value)** 줍니다. A를 때려도 남남인 B는 아무 타격이 없습니다.

#### 🚨 2. 문제와 함정 (AI의 쓰레기 코드)
AI에게 "총알이랑 몬스터 좀 만들어줘!" 했더니 아무 생각 없이 `class`(유리접시) 로 전부 묶어왔습니다.

```csharp
// AI의 치명적 설계 실수 1: 휘발성 총알 데이터를 Class로 묶어 잔여물(렉) 폭발
public class BulletData { public float damage; }

// AI의 치명적 설계 실수 2: 몬스터 스탯을 Class로 묶어서 쪽지 저주 유발
public class MonsterStat { public float hp = 100f; }

public void SpawnMonsters()
{
    MonsterStat baseStat = new MonsterStat();
    
    // 앗! 실제 체력을 복사한 게 아니라 '3번 테이블 쪽지'만 복사됨!
    MonsterStat goblinA = baseStat; 
    MonsterStat goblinB = baseStat; 
    
    // 고블린 A를 한 대 때렸는데... 똑같은 쪽지를 본 고블린 B의 체력도 같이 50으로 줄어드는 대참사!
    goblinA.hp -= 50f; 
}
```

**🔍 잠깐! 코드 외계어(C# 문법) 독해 가이드**
디자이너님, 코드는 결국 일정한 공식으로 쓰이는 문장입니다.

**[첫 번째 문장 독해]** `public void SpawnMonsters()`
*   `public` = 누구나 외부에서 이 기능을 실행할 수 있게 자물쇠를 열어둠 (전역).
*   `void` = **(🐶 강아지 훈련 비유)** 강아지에게 "공 물어와!" 시키면 내 손에 공(결과물 데이터)을 툭 뱉어주죠? 하지만 "앉아!" 시키면 강아지가 털썩 앉는 '행동'만 하고 내 손에 쥐여주는 물건은 아무것도 없습니다. 이렇게 **"행동만 하고 내 손에 결과물을 뱉어 주지 않아도 돼!"** 라고 컴퓨터에게 미리 알려주는 단어가 바로 텅 비어있다는 뜻의 `void` 입니다.
    *   **❓ 굳이 결과물을 안 뱉게(void) 하는 이유가 뭔가요?** 
        게임에서 일어나는 수많은 일들은 그저 '상태를 바꾸고 끝나는' 일입니다. 예를 들어 "배경음악을 보스 테마곡으로 바꿔라" 거나 "몬스터를 소환해라!" 같은 명령은 스피커의 소리가 바뀌거나 화면에 몬스터가 등장하는 게 목적이지, 행동 후 굳이 나에게 "음악 잘 틀었습니다" 라는 영수증(데이터)을 만들어서 건네줄 필요가 없습니다. 불필요한 데이터를 계속 뱉게 만들면 컴퓨터 메모리만 더러워지기 때문에, **"그냥 세상의 상태만 바꾸고 영수증은 버려!"** 라고 단호하게 낭비를 차단하는 것입니다.
*   `SpawnMonsters()` = "몬스터 소환" 이라는 이름의 **'액션 동작'** 선언.
*   👉 **코드 작성 공식:** `[접근허가범위] [돌려줄결과물타입] [자유롭게지은이름]()`
*   🗣️ **완벽하게 소리내어 읽기:** *"이 액션은 누구나 외부에서 마음대로 누를 수 있고(`public`), 이 행동이 끝나더라도 나한테 따로 계산된 데이터를 뱉어내지는 않아(`void`). 이 액션 묶음의 이름은 `SpawnMonsters` 라고 지을게!"*

**[두 번째 문장 독해]** `MonsterStat baseStat = new MonsterStat();`
*   `MonsterStat` = 규격 강제. "여기는 몬스터 스탯만 담을 수 있는 전용 상자야!"
*   `baseStat` = 디자이너가 마음대로 지어서 붙인 '상자의 이름표'입니다.
*   ` = new MonsterStat();` = 비어있는 원본 상자를 새로 뚝딱 만들어서(`new`), 저 이름에 연결하라.
*   👉 **코드 작성 공식:** `[상자규격] [이름표명칭] = new [상자규격]();`
*   🗣️ **완벽하게 소리내어 읽기:** *"앞으로 `MonsterStat` 구멍만 뚫려 있는 전용 상자를 사용할 건데, 지금 비어있는 새 상자를 하나 제작해서(`new`), 거기다가 `baseStat` 이라는 이름표 마스킹 테이프를 딱 붙여 놔!"*

**🤔 왜 하필 `class`라고 쓰면 지옥이 열릴까요?**
컴퓨터에게 `class`는 **"장기 보관용 VIP 손님(Heap 메모리)"**입니다. 수만 개의 빗자루 전용 쓰레기가 쌓이는 것은 둘째 치고, VIP라서 함부로 복사도 안 해줍니다. 그냥 위치(Reference) 쪽지만 틱 던져줍니다. 개발자들이 "어? 나는 A만 때렸는데 왜 화면 전체 몬스터 체력이 깎여요?" 하면서 밤새워 머리를 쥐어뜯게 만드는 대표적인 주범입니다. 

#### ✅ 3. 모범 답안 (종이접시의 기적)
단순 수치 계산용 데이터나 수만 번 날아가는 총알은 반드시 `struct`(종이접시) 라벨로 바꿔야 합니다.

```csharp
// 디렉터의 교정: 메모리 찌꺼기가 생성 안되고 완벽히 독립 복사되는 Struct 사용
public struct BulletData { public float damage; }
public struct MonsterStat { public float hp; }
```

#### 🗣️ 4. 지휘관의 프롬프팅 (이렇게 명령하세요)
> **"AI야, 총알이나 몬스터 스탯을 왜 `class` (참조 타입)로 짜서 GC 렉이랑 주소 공유 버그를 유발하냐? 당장 GC 쓰레기가 발생 안 하고 안전하게 독립 복사(Value)만 되도록 싹 다 `struct` 타입으로 뜯어고쳐!"**

#### 📝 5. 복습 및 과제
* **아이큐 테스트:** 절대로 파괴되지 않고 **'오직 한 마리만 존재하며 게임 내 수많은 개체들이 이 녀석의 위치 쪽지 번호를 봐야 하는' 보스 몬스터의 방 데이터**는 조심스럽게 모셔두는 유리접시(`class`)여야 할까요, 툭하면 자기 모습을 복제해버리는 종이접시(`struct`)여야 할까요? 정답을 생각하며 다음으로 넘어가세요!

### 📅 [Day 2] 매직 스트링(Magic String)과 키오스크 주문

![말로 외치는 혼란스러운 식당(String)과 완벽히 통제된 네온 키오스크(Enum)의 대비 이미지](/assets/images/enum_kiosk.png)

#### 📖 1. 예습 (식당 키오스크 비유)
> 🍔 **말로 주문하기(String) vs 키오스크 번호표(Enum)**
> 알바생에게 말로 "치즈버거 하나요!" 라고 주문(`string`)하면 어떤 일이 벌어질까요? 
> 주방장이 "치주버거?", "치즈바거?" 처럼 오해할 확률이 생깁니다. 컴퓨터 세상에서 대소문자를 섞어 쓰면 컴퓨터는 **없는 메뉴라며 통신을 단절합니다.**
> 
> 반면, 메뉴판의 **[2번: 치즈버거 세트]** 버튼(키오스크)을 누르고 2번이라고 주문(`enum`)하면 어떨까요? 주방장은 무조건 2번 레시피를 만들기 때문에 소통에 오해가 낄 확률이 **0%**가 됩니다. 유니티에서 `enum`은 절대로 오타가 날 수 없는 '메뉴판 번호표' 역할을 합니다.

#### 🚨 2. 문제와 함정 (AI의 쓰레기 코드)
AI에게 몬스터의 상태(대기, 공격, 죽음) 관리를 짜달라고 했더니, 멍청하게 말로 주문하는 구조(`string`)로 짜왔습니다.

```csharp
public class MonsterLogic : MonoBehaviour 
{
    private string currentState = "Die"; // 첫 글자가 대문자 D

    void Update() 
    {
        // 끔찍한 오타 버그: 밑에서는 소문자 'd'ie 를 찾고 있음!!
        if (currentState == "die") 
        {
            Destroy(gameObject);
        }
    }
}
```
*   **❓ 그냥 기획자나 프로그래머가 오타 안 내려고 조심해서 쓰면 안 되나요?**
    *   초보 코더들은 '조심'해서 버그를 막으려 합니다. 하지만 아키텍처 디렉터는 **인간이 아예 실수조차 할 수 없도록 시스템적으로 막아버립니다.** 게임 코드 15만 줄을 사람이 매번 눈으로 검수할 순 없기 때문입니다.

#### ✅ 3. 모범 답안과 외계어 독해
상태값은 반드시 키오스크 번호표인 `enum`(열거형)으로 강제해야 합니다. 아래의 완벽한 구조를 보며 C# 외계어를 디자이너의 언어로 독해해 봅시다.

```csharp
// [문장 1 독해] 키오스크 메뉴판(Enum)을 먼저 정의합니다.
public enum MonsterState { Idle, Attack, Die }

public class MonsterLogic : MonoBehaviour 
{
    // [문장 2 독해] 문자열이 아닌 메뉴판(Enum) 전용 변수를 만듭니다.
    private MonsterState currentState = MonsterState.Die; 

    void Update() 
    {
        // [문장 3 독해] 오타가 날 확률 0%. 유니티 시스템이 자동 완성을 강제합니다.
        if (currentState == MonsterState.Die) 
        {
            Destroy(gameObject); // [문장 4 독해] 오브젝트 파괴
        }
    }
}
```

**🔍 잠깐! 코드 외계어(C# 문법) 독해 가이드**
디벨로퍼의 외계어 코드를 의미 단위로 뜯어봅시다.

**[문장 1]** `public enum MonsterState { Idle, Attack, Die }`
*   👉 **코드 공식:** `[전역허용] [메뉴판(enum)] [메뉴판이름] { 버튼1, 버튼2, 버튼3 }`
*   🗣️ **소리내어 읽기:** *"이 게임 어디서나 쓸 수 있게(`public`), '몬스터 상태(`MonsterState`)'라는 이름의 새로운 키오스크 메뉴판(`enum`)을 하나 만들 건데, 그 메뉴 버튼은 대기, 공격, 죽음 딱 3개만 세팅해 둬!"*

**[문장 2]** `private MonsterState currentState = MonsterState.Die;`
*   👉 **코드 공식:** `[은닉성] [규격(메뉴판타입)] [특정주문서이름] = [초기설정값];`
*   🗣️ **소리내어 읽기:** *"앞서 만든 키오스크 메뉴판 규격(`MonsterState`)만 찍을 수 있는 '빈 주문서'를 하나 줘. 그 주문서의 이름은 내 마음대로 `currentState`라고 부를게. 그리고 게임이 시작될 때 이 주문서엔 기본값으로 아까 그 메뉴판에 있는 `Die` 버튼을 꾹 눌러서(`= MonsterState.Die`) 담아 놔! 아, 남들이 이 주문서 내용을 함부로 조작 못 하게 자물쇠(`private`) 채우는 거 잊지 말고."*
*   **❓ 메뉴판(문장 1)을 만들었으면 끝난 거 아닌가요? 대체 왜 '전용 변수(주문서)'를 또 새로 만드나요?**
    *   "이래서 이걸 쓴 거에요~" 하고 서술해 드리자면 이렇습니다! 식당 벽에 **'메뉴판(문장 1: enum)'**을 예쁘게 걸어놨다고 해서, 주방장이 알아서 요리를 시작할까요? 아니죠! 손님이 실제로 어떤 상태(메뉴)인지 적어둘 보관용 **'실물 주문서 종이(변수)'**가 하나 필요한 겁니다. 
    *   그래서 "저기요! 방금 벽에 건 저 메뉴판(`MonsterState`) 룰에 맞춰 작성할 수 있는 나만의 주문서 용지(`currentState`) 하나 갖다주세요!" 라고 컴퓨터에게 내 몬스터의 상태를 메모할 빈 종이를 별도로 요구하는 것입니다. 이 종이가 없으면 몬스터는 현재 자기가 대기 중인지 공격 중인지 적어둘 물리적인 메모장 자체가 없는 거니까요!

**[문장 3과 4]** `if (currentState == MonsterState.Die) { Destroy(gameObject); }`
*   👉 **코드 공식:** `만약 (비교) { 맞다면 이거실행; }`
*   🗣️ **소리내어 읽기:** *"만약(`if`) 상자 안에 들어있는 현재 상태가 진짜로 메뉴판의 `Die` 버튼이랑 완벽히 일치한다면(`==`), 묻지도 따지지도 말고 이 게임오브젝트를 당장 화면에서 파괴시켜버려(`Destroy`)!"* 
*   **❓ 파괴(`Destroy`)하면 어제 배운 가비지 쓰레기(GC)가 지워지는 건가요?**
    *   지휘관님, 굉장히 날카로운 눈썰미입니다! 정답만 말씀드리면 **'아닙니다'**. `Destroy`는 화면에서 안 보이게 오브젝트를 지울 뿐, 오히려 뒤에선 청소부들이 치워야 할 **쓰레기(GC)가 왕창 발생합니다!** 
    *   그래서 똑똑한 디렉터들은 수시로 터지는 총알 같은 건 파괴(`Destroy`) 대신 그냥 눈동자만 끄고 켜는 레이어 처리(`SetActive(false)`)를 해서 평생 돌려쓰는 **'오브젝트 풀링'(Month 6 과정)** 기법을 씁니다. 당장은 "화면에서 지워라!" 정도로만 독해하시면 충분합니다.

#### 🗣️ 4. 지휘관의 프롬프팅
> **"AI야, 몬스터 상태를 `string` 문자열로 하드코딩해서 나중에 좀비 버그 터트릴 셈이냐? 당장 `public enum MonsterState` 서브 메뉴판부터 먼저 생성해서 대소문자 오타 에러가 절대 나지 않게 (Type Safe) 구조 갈아엎어!"**

### 📅 [Day 3] Resources.Load의 저주와 통제권 탈환

#### 📖 1. 예습 (길 찾기 비유)
> 🗺️ **약도 그려주기 vs 물건 직접 쥐여주기**
> 
> 팀원에게 폭탄 라이터를 주면서 "우리 집 2층 서재 3번째 서랍에 있어(`Resources.Load("Floor2/Drawer3/Lighter")`)" 라고 위치(경로 텍스트)로만 알려줍니다. 나중에 집 청소를 하다가 거실로 라이터를 옮기면? 팀원은 서재 3번째 서랍만 열어보고 **"폭탄 없는데요!! 뻗겠습니다!!"** 라며 게임을 통째로 멈춰버립니다.
> 
> 하지만, 아예 폭탄 라이터 자체를 팀원의 손에 **직접 쥐여준다면(`[SerializeField]`)** 어떨까요? 청소를 해서 집 주소가 바뀌든 거실로 가든, 팀원의 꽉 쥔 손에는 폭탄 라이터가 안전하게 들려있습니다. 절대로 길을 잃지 않죠!

#### 🚨 2. 문제와 함정 (아티스트의 통제권을 뺏는 짓)
AI에게 몬스터가 죽을 때 이펙트를 터트리라고 시켰더니, 이펙트 경로를 텍스트로 하드코딩해버렸습니다.

```csharp
public void OnDeath()
{
    // 아티스트가 에셋 폴더 위치만 바꿔도 게임 전체 이펙트가 다운되는 끔찍한 경로 의존성 구조
    GameObject effect = Resources.Load<GameObject>("Prefabs/VFX/DieEffect");
    Instantiate(effect, transform.position, Quaternion.identity);
}
```
**🤔 디자이너/아티스트의 극한 분노 지점**
저 이펙트 파일(`DieEffect.prefab`)을 조금 더 정리하기 위해 유니티에서 폴더를 `Prefabs/Monsters/Boss/VFX/` 로 마우스로 옮기는 순간, 프로그래머가 짠 코드는 옛날 텍스트 주소만 기억하고 있기 때문에 파일을 못 찾고 빨간색 에러를 내뱉습니다. 아티스트는 자기가 폴더 하나 정리해 놓고도 프로그래머에게 소스를 고쳐달라며 빌어야 합니다.

#### ✅ 3. 모범 답안 (인스펙터 슬롯 방어법)
경로는 절대 타이핑으로 치면 안 됩니다. 인스펙터(Inspector) 창에 물리적인 '빈 공간(드래그 앤 드롭용 구멍)'을 뚫어서 아티스트에게 권한을 줘야 합니다.

```csharp
public class MonsterLogic : MonoBehaviour 
{
    // [문장 1 독해] 인스펙터에 아티스트용 슬롯 뚫기
    [SerializeField] private GameObject dieEffectPrefab; 

    public void OnDeath()
    {
        // [문장 2 독해] 폴더를 옮겨도 절대 연결이 끊어지지 않음
        Instantiate(dieEffectPrefab, transform.position, Quaternion.identity);
    }
}
```

**🔍 잠깐! 코드 외계어(C# 문법) 독해 가이드**

**[문장 1 독해]** `[SerializeField] private GameObject dieEffectPrefab;`
*   👉 **코드 작성 공식:** `[창문뚫기] [은닉성] [규격(게임물체)] [이름표];`
*   🗣️ **완벽하게 소리내어 읽기:** *"이 변수는 다른 놈들 코드가 못 건드리게 꽁꽁 숨기고 자물쇠를 걸어(`private`). 대신, 특별히 유니티 화면(Inspector)에만 보이도록 투명 창문 구멍 하나만 예쁘게 뚫어줘(`[SerializeField]`)!! 우리 아티스트가 이펙트 오브젝트(`GameObject`)를 마우스로 직접 끼워 넣을 수 있어야 하니까, 그 구멍 이름은 `dieEffectPrefab` 이야!"*
*   **❓ 어차피 에디터 화면에 보이게 할 거면, 쿨하게 `public`으로 선언하면 안 되나요?**
    *   이게 바로 AI나 어설픈 코더들이 매일같이 저지르는 실수입니다. 화면에 보이게 하려고 `public`을 걸어버리면, 우리 팀 아티스트뿐만 아니라 **다른 기능들의 무수한 스크립트 코드까지 이 변수를 마음대로 바꾸고 조종할 수 있게 대문이 박살나 버립니다.** 
    *   그래서 코더의 침입은 완벽히 차단하고(`private`), 유니티 엔진(눈)에만 보이게 유리창만 내주는 고급 스킬이 바로 `[SerializeField]`입니다. 즉 **"코드 접근은 거부하되 에디터 편집 권한만 열어주는 완벽한 아키텍처"**입니다.

#### 🗣️ 4. 지휘관의 프롬프팅
> **"AI야, 이펙트 불러올 때 구식으로 `Resources.Load` 써서 하드코딩 치지 마! 아티스트가 폴더 정리하면 게임 터지잖아. 당장 `[SerializeField] private GameObject dieEffectPrefab` 선언해서 인스펙터 창으로 뚫어놔. 내가 직접 드래그 앤 드롭 해서 끼워 넣고 평생 폴더 맘대로 바꿀 거니까!"**

### 📅 [Day 4] 무지성 Public 개방의 함정과 은닉화

#### 📖 1. 예습 (투명 저금통 비유)
> 💰 **누구나 털어가는 공용 저금통(`public`) vs 은행 창구(`private` + `Property`)**
> 
> 우리 몬스터의 체력을 **누구나 집어갈 수 있는 공용 저금통(`public hp`)**으로 열어두면 어떻게 될까요? 사운드 매니저, 이펙트 매니저가 와서 동전을 빼가도 아무도 막지 않습니다. 나중에 버그가 터져서 체력이 마이너스가 됐을 때, 수만 줄의 코드 중 대체 "어떤 놈"이 몬스터 체력을 건드렸는지 범인을 잡을 수가 없습니다.
> 
> 반면, 체력을 **금고(`private`)**에 가둬두고, 앞에는 **투명한 조회 창구(`Property`)**만 만들어두면 어떨까요? 다른 녀석들은 유리창 너머로 현재 체력이 얼마인지만 확인(조회)할 수 있고, 체력을 깎는 건 오직 금고 안에 있는 '몬스터 본인 스스로'만 할 수 있습니다. 완전한 통제 시스템이죠!

#### 🚨 2. 문제와 함정 (AI의 쓰레기 코드)
AI에게 몬스터 코드를 짜보라고 하면, 거의 100% 무지성으로 저금통 뚜껑을 까버립니다.

```csharp
public class MonsterLogic : MonoBehaviour 
{
    // 심각한 설계 결함: 아무 스크립트나 와서 피를 조작해 버릴 수 있음!
    public int hp = 100;
}
```
**🤔 이게 왜 지옥을 부르나요?**
만약 게임 후반부에 아이템을 먹는 코드가 꼬여서 실수로 몬스터의 `hp`를 0으로 바꿔버린다면? 몬스터는 아이템을 먹었는데 갑자기 돌연사합니다. 프로그래머는 몬스터 스크립트만 백날 뒤지며 "왜 죽지? 체력 로직에 문제없는데?" 하고 며칠 밤을 새웁니다. 무단 침입 범인이 엉뚱한 '아이템 코드'에 있다는 걸 상상도 못 하는 거죠! 이게 바로 `public` 남용이 부르는 나비효과입니다.

#### ✅ 3. 모범 답안 (은닉화와 Property 창구)
변수는 무조건 `private`으로 꽁꽁 잠그고, 조회를 위한 '읽기 전용 창구(Property)'만 열어주는 것이 아키텍처의 기본입니다.

```csharp
public class MonsterLogic : MonoBehaviour 
{
    // [문장 1 독해] 유리창은 열어두고 설정은 막는 기적의 문법
    public int Hp { get; private set; } = 100;

    public void TakeDamage(int damage)
    {
        // 체력은 오직 '나 자신' 안에 있는 함수 안에서만 깎을 수 있다!
        Hp -= damage;
    }
}
```

**🔍 잠깐! 코드 외계어(C# 문법) 독해 가이드**

**[문장 1 독해]** `public int Hp { get; private set; }`
*   👉 **코드 작성 공식:** `[접근허용] [규격(정수)] [이름] { 열람허락; 수정은자물쇠; }`
*   🗣️ **완벽하게 소리내어 읽기:** *"이 `Hp`라는 녀석은 누구나 밖에서 유리창 너머로 쳐다볼 수 있어(`public`, `get`). 하지만 그 안에 있는 수치를 조작하고 수정하는 건 자물쇠로 꽉 잠가놔서(`private set`) 외부의 그 누구도 절대 못 만져! 체력은 몬스터 오직 나 자신만 깎을 수 있다!"*
*   **❓ 어차피 못 건드리게 할 거면 그냥 전부 다 통째로 숨겨버리면(`private`) 안 되나요?**
    *   완벽하게 자물쇠만 걸어 숨겨버리면, 나중에 화면에 UI를 그려주는 스크립트가 몬스터의 남은 체력이 얼마인지 볼 수가 없어서 **'머리 위 체력 빨간 바'**를 백지상태로 놔두게 됩니다! 
    *   그래서 **"눈으로 보는 것(`get`)은 체력바 UI를 위해 온 세상에 공개하되, 값을 조작하는 것(`set`)은 철저히 비밀로 잠근다(`private`)"**는 이 위대한 타협점인 프로퍼티(Property)가 등장한 것입니다.

#### 🗣️ 4. 지휘관의 프롬프팅
> **"AI야, 몬스터 스탯을 왜 무지성 `public`으로 열어서 여기저기 다른 스크립트가 다 쑤시고 다니게 냅둬? 나중에 버그 추적 어떻게 하려고! 당장 `{ get; private set; }` 달아서 외부에서는 남은 체력 읽기만 가능하게 철저하게 은닉화(Encapsulation) 시켜!"**
